# 架构设计文档

## 设计背景

团队面临一个新场景：产品用 AI 快速搭建平台生成原型代码，开发需要将其迁移到正式项目。最初用 3 个独立 skill 分步处理（迁移、完善、清理），暴露了以下问题：

1. 每天 token 消耗 100M，大量浪费在重复读取和循环验证
2. 3 个 skill 需要手动依次触发，无法断点恢复
3. 功能遗漏只能靠人工自查，没有自动化验证
4. skill 文件绑定了特定项目路径，无法复用

## 架构决策

### 决策 1：合并为单一入口 + 分阶段 reference

**问题**：3 个 skill 各自独立，重复扫描代码，上下文互不共享。

**方案**：合并为 1 个 SKILL.md 入口 + 6 个 reference 文件，按需加载。

**权衡**：
- 单文件方案（600 行全塞一起）→ 每次触发加载 6000 token，浪费
- 多 skill 方案（保持 3 个独立 skill）→ 无法共享中间产物，重复工作
- 最终方案（入口 + references）→ 入口 ~150 行 ≈ 1500 token，按需加载对应阶段

### 决策 2：TokenGuard 内嵌而非独立 Agent

**问题**：token 消耗过高，需要控制。

**方案**：将 token 节省策略作为规则文档内嵌到工作流中，而非独立的 Agent 角色。

**原因**：
- 独立 Agent 需要额外的上下文切换开销
- token 控制是约束条件，不是独立任务
- 作为 reference 文件加载，每个阶段都能遵守

### 决策 3：状态持久化到 .agent/ 目录

**问题**：长对话容易中断（context 超限、网络断开），需要断点恢复。

**方案**：所有中间产物和进度状态写入 `.agent/` 目录的 JSON 文件。

**产物设计**：
```
.agent/
├── project_config.json   → 项目配置（跨会话持久）
├── project_spec.md       → 项目规范（首次生成）
├── status.json           → 进度追踪（哪个阶段完成了）
├── codebase_map.json     → 原型全景图（Phase 1 产出，Phase 4 消费）
├── task_list.json        → 任务列表（Phase 1 产出，Phase 1-3 消费）
├── file_digest.json      → 文件摘要缓存（TokenGuard 核心）
├── change_log.json       → 变更追踪（增量验证用）
└── coverage_report.json  → 覆盖率报告（Phase 4 产出）
```

**好处**：新对话说「继续」→ 读 status.json → 从断点恢复，不从头来。

### 决策 4：首次配置自动读取目标项目规范

**问题**：最初 zlj-spec.md 是手写的静态文件，绑定了特定项目，同事无法复用。

**方案**：首次使用时自动读取目标项目的 eslintrc / CLAUDE.md / package.json / 目录结构，生成 project_spec.md。

**流程**：
```
询问路径 → 自动读取规范文件 → 生成 spec → 问用户补充约束 → 保存
```

**好处**：跨项目通用，无需手写 spec，同事拿到即可用。

### 决策 5：功能回测作为独立阶段

**问题**：重构后功能遗漏只能靠人工自查。

**方案**：新增 Phase 4 Verifier，从 codebase_map.json 提取原型功能清单，逐项比对目标代码。

**设计**：
- 不重新扫描原型代码（复用 Phase 1 的 codebase_map）
- 优先从 file_digest.json 读取目标代码信息（不全量读文件）
- 输出 coverage_report.json，missing 项自动补全
- 最多 2 轮补全，避免无限循环

### 决策 6：智能模式判断

**问题**：用户说「重构」但目标已有大量代码，全量覆盖会破坏已有工作。

**方案**：触发后先检查目标目录文件数，≥5 个组件文件时自动建议降级为增量/回测模式。

**原则**：宁可多问一句，不要默默覆盖。

## TokenGuard 10 条策略设计思路

| 策略 | 解决的浪费场景 | 节省幅度 |
|------|--------------|---------|
| 1. 读一次缓存摘要 | 同一文件被多个阶段重复读取 | ~30% |
| 2. 分段读取（120行阈值） | 大文件全量读入但只用几行 | ~15% |
| 3. 增量验证 | Review 时全量扫描所有文件 | ~20% |
| 4. 批量修复 | 发现一个问题修一个再扫一遍 | ~15% |
| 5. 输出精简 | 向用户复述已写入的代码 | ~5% |
| 6. hash+mtime 跳过 | 未变更文件被重新读取 | ~10% |
| 7. 写入前不重读 | Write 前多一次确认性 Read | ~5% |
| 8. Grep 代替 Read | 只需确认存在性却读整个文件 | ~10% |
| 9. 并行工具调用 | 独立查询串行执行浪费轮次 | ~5% |
| 10. 禁止探索性读取 | 无目的浏览文件 | ~10% |

综合节省：实测回测模式 ~70%（15K vs 预估 50-80K）。

## 演进路线

| 版本 | 状态 | 内容 |
|------|------|------|
| v1 | 已完成 | 3 个独立 skill，手动串联 |
| v2 | 已完成 | 合并为 Pipeline + TokenGuard + 回测 |
| v3 | 已完成 | 通用化（去硬编码 + 首次自动配置） |
| v4 | 待验证 | 全流程实测数据（新分支跑一次） |
| v5 | 规划中 | 支持 Vue/其他框架的原型（当前仅验证了 React） |

## 局限性

1. **依赖 Claude Code Skill 框架** — 不是独立工具，必须在 Claude Code 环境下运行
2. **prompt 驱动** — 策略靠文字约束，Claude 不一定 100% 遵守（特别是 TokenGuard）
3. **单会话上下文限制** — 全流程如果原型很大，可能需要多次「继续」
4. **未验证非 React 项目** — 当前只在 React + JSX 项目上验证过
